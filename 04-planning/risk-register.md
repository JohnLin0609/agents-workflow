# 風險登記簿(Risk Register)

> **目的**:把已識別的風險記下、評估、追蹤、緩解。
> **負責人**:PM + 技術 lead
> **Review**:每週 / 每 sprint review 一次
> **核心原則**:這是「活文件」,不是寫完就丟。沒被定期更新的風險登記簿等於沒寫。

---

## 1. 評分標準

### 影響(Impact)
| 等級 | 描述 |
|------|------|
| 1 - 輕微 | 局部延誤、可吸收 |
| 2 - 中等 | 部分功能延後或品質下降 |
| 3 - 重大 | 里程碑延誤、需重大調整 |
| 4 - 嚴重 | 專案失敗 / 嚴重商業損失 |

### 機率(Probability)
| 等級 | 描述 |
|------|------|
| 1 - 很低 | < 10% |
| 2 - 低 | 10–30% |
| 3 - 中 | 30–60% |
| 4 - 高 | > 60% |

### 風險分數
- 分數 = 影響 × 機率
- 1–4:低 / 5–9:中 / 10–16:高(必須有緩解計畫)

## 3. 風險詳述(高分數風險)

### R-001:金流商審核延遲

**描述**
金流商商務審核流程通常需 2–4 週,若卡關會直接影響上線日期。

**早期徵兆**
- 審核回覆超過 3 個工作天
- 補件要求超過 2 次

**緩解策略**
1. 第一週就遞交申請,不要等開發完成
2. 同時申請第二家金流商作為備案
3. 與金流商建立直接窗口

**應變計畫(若發生)**
- 若延遲 > 2 週,評估改用備案金流商
- 與 stakeholder 溝通延期可能

**負責人**:PM
**Review 頻率**:每週

## 5. 風險類別參考

> 開始識別時可以從這些類別檢查,避免漏。

- **技術**:技術不成熟、效能瓶頸、整合困難、技術債
- **人力**:人員離職、技能不足、團隊衝突
- **時程**:估算過於樂觀、依賴延遲、範圍蔓延
- **外部依賴**:第三方服務、API 變更、供應商審核
- **合規**:法規變更、個資、跨境
- **商業**:需求變更、市場改變、競爭
- **維運**:部署失敗、災難復原、安全事件
