# Code Review Checklist

> **目的**:讓 reviewer 不漏看重點,讓被 review 者知道會被看什麼。
> **使用方式**:可貼進 PR template,或團隊 wiki 連結。
> **核心原則**:**Review 的是程式碼,不是人。**對事不對人,中性語氣。

---

## Review 心態

- 假設作者已盡力,有疑問先問「為什麼這樣設計?」再評論
- 區分:**必須改**(blocking)/ **建議改**(suggestion)/ **想法分享**(nit, optional)
- 一次看完,別擠牙膏式分多次評論
- 大 PR 拒收 → 請作者拆分,而不是硬看
- 24h 內回覆,避免卡住節奏

---

## ✅ 通用檢查項目

### 1. 正確性
- [ ] 程式碼真的實作了 PR 描述的功能
- [ ] 對應的需求 / 票號驗收標準都涵蓋
- [ ] 邊界條件有處理(空、null、極大/極小、並發)
- [ ] 錯誤情境有處理,不會默默吞掉

### 2. 設計與架構
- [ ] 符合 design 文件,沒有偏離
- [ ] 沒有越界:不該知道的東西沒被存取
- [ ] 沒有循環依賴
- [ ] 沒有過度設計(YAGNI 違反:抽象只有一個實作?)
- [ ] 沒有重複程式碼可抽出(DRY,但不過度)

### 3. 可讀性
- [ ] 命名清楚,符合 coding standards
- [ ] 函式 / 類別不過長
- [ ] 巢狀不過深(早返回?)
- [ ] 註解解釋「為什麼」,不是「做什麼」
- [ ] 沒有死碼、註解掉的舊碼

### 4. 測試
- [ ] 有對應的測試
- [ ] 測試涵蓋 happy path + 主要邊界
- [ ] 測試命名清楚(看名字知道在測什麼)
- [ ] 測試獨立,不依賴順序
- [ ] 沒有 flaky test(時間、隨機、外部依賴未 mock)

### 5. 錯誤處理 & 日誌
- [ ] 錯誤訊息對使用者友善、對工程師有用
- [ ] Log 等級合理(不是全 INFO 也不是全 ERROR)
- [ ] **沒有 log 敏感資料**(密碼、token、PII、卡號)
- [ ] 有 request_id 或追蹤標記

### 6. 安全
- [ ] 輸入驗證(別信任前端)
- [ ] 權限檢查(每個操作驗證 ownership)
- [ ] SQL 用參數化查詢
- [ ] 沒有 hardcoded secret
- [ ] 第三方依賴沒有已知漏洞

### 7. 效能
- [ ] 沒有 N+1 查詢
- [ ] 大集合操作有考慮分批 / 分頁
- [ ] 同步阻塞操作合理(沒卡住 event loop)
- [ ] 慎重使用 cache,有失效策略

### 8. 文件與相容性
- [ ] API spec / Schema 已同步更新
- [ ] README / 設定文件已更新
- [ ] 向後相容(不破壞舊版 client)
- [ ] DB migration 有 rollback 計畫

### 9. Git / PR 衛生
- [ ] Commit message 符合規範
- [ ] PR 一個目的,沒有混雜
- [ ] 沒有不相關的檔案被改到
- [ ] CI 通過(lint、test、build)

---

## 🚨 紅線(發現直接 block)

- 在 log 印敏感資料
- Hardcoded secret 在程式碼中
- SQL 字串拼接(injection 風險)
- 沒有測試的關鍵業務邏輯
- 直接讀寫 prod DB 的腳本
- 移除既有功能但沒人簽核

---

## 💬 評論用語建議

| 情境 | ✓ 建議 | ✗ 避免 |
|------|--------|--------|
| 必改 | "這裡有 SQL injection 風險,需要改用參數化查詢" | "你怎麼會寫成這樣" |
| 建議 | "考慮抽出成獨立函式,X 模組也用得到" | "這寫得好醜" |
| 疑問 | "好奇為什麼這裡用 setTimeout?" | "這明顯有問題吧" |
| 讚賞 | "這個錯誤處理寫得很周到 👍" | (不只挑錯,也指出好的) |

**前綴慣例**(可選):
- `[blocking]` 必須改
- `[suggestion]` 建議改
- `[nit]` 雞蛋裡挑骨頭,可不改
- `[question]` 純粹問問
- `[praise]` 讚賞

## 特殊類型 PR 額外檢查

### DB Migration
- [ ] Rollback script 有寫
- [ ] 大表變更評估鎖表時間
- [ ] 向後相容(舊程式還能跑)

### API 變更
- [ ] 版本策略有遵守
- [ ] Breaking change 已公告
- [ ] 文件已更新

### 安全相關
- [ ] 至少一位 security owner review
- [ ] 對應 threat model 文件

### 效能相關
- [ ] 有 benchmark 數據佐證
- [ ] 沒有顯著回歸
