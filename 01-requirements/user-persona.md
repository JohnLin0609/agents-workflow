# 使用者輪廓(User Persona)

> **目的**:讓團隊對「我們在為誰設計」有共同的具體想像,而不是抽象的「使用者」。
> **負責人**:PM / UX
> **建議數量**:1–3 個主要 persona,不要超過 5 個

---

## Persona 1:[名字 + 一句話定位]
> 例:「客服小美 — 每天處理 80 通客戶詢問的資深客服」

### 基本資料
| 項目 | 內容 |
|------|------|
| 名稱 | 小美 |
| 年齡 / 角色 | 32 歲,資深客服 |
| 技術熟悉度 | 中等 |
| 使用裝置 | 桌機為主、偶爾手機 |
| 使用頻率 | 每天 8 小時 |

### 目標(Goals)
> 她想達成什麼?
- 快速回應客戶
- 把複雜問題處理好

### 痛點(Pain Points)
> 她現在卡在哪裡?
- 重複問題太多,佔用時間
- 找資料要切換多個系統

### 行為與情境(Context)
> 她什麼時候、在什麼情境下會用到我們的系統?
- 早上 9 點上線,先看當日待處理清單
- 同時開 3–4 個對話視窗

### 引言(Quote)
> 一句能代表這個人的話。
> 「我不怕難的問題,我怕一整天都在回同樣的問題。」

### 設計啟示(Design Implications)
> 從這個 persona 我們應該重視什麼?
- 介面要支援快速切換多對話
- 常見問題要能一鍵套用
