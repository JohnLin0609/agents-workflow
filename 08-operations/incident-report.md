## 4. 根本原因分析(Root Cause Analysis)

### 4.1 直接原因
DB 連線池上限設定為 20,在流量高峰時無法滿足需求,新請求等待連線超時,造成大量 5xx。

### 4.2 為什麼會發生?(5 Whys)

1. **為什麼下單失敗?** → DB 連線等待超時
2. **為什麼連線等待超時?** → 連線池滿
3. **為什麼連線池滿?** → 每個 pod 池上限 20,在 2x 流量下不夠
4. **為什麼上限設 20?** → 半年前初版設定,沒隨流量增長調整
5. **為什麼沒調整?** → 沒有監控覆蓋這個指標,沒人注意

### 4.3 貢獻因素(Contributing Factors)

不只一個原因。完整來說:
- 連線池上限沒隨流量調整(主因)
- 連線池使用率沒監控(讓問題不可見)
- 行銷活動沒做 pre-load 測試
- 告警觸發後,初次嘗試 scale up 沒效(沒人意識到 bottleneck 是 DB 端)

### 4.4 為什麼沒有更早被發現?
- 連線池使用率沒被監控
- 沒有預警(80% 使用率時)
- 過去流量沒到過這個級別

## 8. 經驗教訓(Lessons Learned)

> 可以放進 lessons-learned.md 累積。

- **教訓 1**:資源使用率(不只是「夠不夠」,還有「在用多少」)必須監控。我們監控了「服務是否健康」,但漏掉了「離極限多遠」。
- **教訓 2**:Scale up 服務不等於 scale up 容量。要看整條鏈的瓶頸。
- **教訓 3**:行銷與工程之間的「上線前溝通」流程要正式化。
