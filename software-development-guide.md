## 1. 程式架構

### 先理解問題,再談架構

最大的錯誤是先選技術或框架,再硬套到需求上。實際順序應該反過來:先釐清這個系統要解決什麼問題、預期多少使用者、資料量多大、未來可能怎麼變、團隊有多少人、效能與一致性哪個優先。沒有這些資訊就開始畫架構圖,基本上是憑感覺。

### 核心原則

不論大小專案,有幾個原則幾乎都適用:

- **關注點分離(Separation of Concerns)**——把不同職責的程式碼分開,不要讓 UI 邏輯混在資料存取裡。
- **高內聚低耦合**——同一個模組內的東西應該緊密相關,不同模組之間的依賴要少而清楚。
- **依賴方向**——讓抽象層被具體實作依賴,而不是反過來(SOLID 中的 D,也是 Clean Architecture 的核心)。
- **單一資料源(Single Source of Truth)**——同一份資料不要散落在多處被各自更新。

### 選擇架構風格

常見的幾種風格,各有適用場景:

- **分層架構(Layered)**——最常見,把系統切成表現層、業務邏輯層、資料存取層,適合 CRUD 為主的中小型系統。
- **六角形 / Clean Architecture**——把業務邏輯放在最內層,外面圍著各種介面(DB、API、UI),適合業務邏輯複雜、需要長期維護的系統。
- **事件驅動(Event-driven)**——用訊息或事件解耦元件,適合非同步、需要擴展的系統。
- **微服務**——把系統切成多個獨立部署的服務,適合大團隊、需要獨立擴展的場景,但複雜度成本很高,小團隊通常不該選。

### 實際的設計步驟

建議順序:

1. 用文字或圖把核心領域模型寫清楚(這個系統裡有哪些「東西」、它們之間什麼關係)
2. 定義主要的使用情境(use case),看資料怎麼流動
3. 切模組邊界,問自己「如果這個模組要換掉,要動到多少其他地方」
4. 最後才選技術棧

### 避免常見陷阱

**過度設計**是新手到中階最容易犯的錯——為了想像中的未來需求加抽象層,結果增加複雜度卻沒帶來價值。反過來,完全不考慮擴展性也會在系統長大後付出代價。

原則:**為當下的需求 + 一步之內可預見的變化** 設計,而不是為三年後的幻想設計。

## 2. 好程式的特性

「好程式」的標準會隨情境變動——一段拋棄式腳本和一個要維護十年的系統,評判標準完全不同。但有一組特性幾乎在所有情境下都會被列出來。

### 核心特性

**正確性(Correctness)**
最基本的一條:程式要做它該做的事,而且在邊界條件、錯誤輸入、異常狀況下也不會崩壞或產生錯誤結果。沒有正確性,其他特性都沒意義。這也是為什麼測試這麼重要。

**可讀性(Readability)**
程式碼被讀的次數遠多於被寫的次數。判斷標準:**六個月後的你,或一個新進團隊的人,能不能不問就看懂這段在做什麼。**

**可維護性(Maintainability)**
需求一定會變。好程式讓改動容易、影響範圍可預測。當你要改 A 功能時,不會需要同時動到 B、C、D。

**可測試性(Testability)**
如果程式很難測,通常代表設計有問題——耦合太緊、副作用太多、依賴沒抽出來。可測試的程式自然會推動你寫出更乾淨的架構。

**效能(Performance)**
夠快就好,不是越快越好。過早最佳化會犧牲可讀性換來不需要的速度。

**健壯性(Robustness)**
面對非預期輸入、網路斷線、磁碟滿、第三方服務掛掉時,程式不會整個倒下。

**安全性(Security)**
不洩漏敏感資料、不信任外部輸入、最小權限原則、敏感操作要驗證。對任何會處理使用者資料或對外開放的系統,這是基本要求,不是加分項。

**可擴展性(Scalability)**
當使用量、資料量、團隊規模成長,程式能不能跟著長大。注意這不等於「一開始就上微服務」——可擴展性是設計選擇的彈性,不是堆技術。

**簡單性(Simplicity)**
這是最被低估、但可能最重要的一個。能用簡單方法解決的,就不要用複雜方法。Rich Hickey 有句名言大意是:簡單不等於容易,簡單是「沒有交織在一起」(not interleaved)。

### 排序的建議

**正確性 > 可讀性 > 簡單性 > 其他**

前三者支撐了後面所有特性。一段不正確的程式再快也沒用、一段沒人看得懂的程式遲早會被改壞、一段過度複雜的程式會讓所有其他特性都變難達成。

## 3. 規劃與 Design Pattern

### 結論先講

**Design pattern 不是用來「規劃」程式的,是用來「描述」程式的。** 把它當成規劃的起點,通常會導致過度設計。

### 規劃程式的真正順序

規劃的核心不是 pattern,是**讓問題本身浮出結構**:

1. **釐清需求與情境**——要解決什麼問題?誰在用?資料怎麼進來、怎麼出去?
2. **找出領域中的「名詞」與「動作」**——寫下這個系統裡有哪些實體、它們之間什麼關係、會發生哪些事件
3. **畫出主要使用流程**——挑 3–5 個最重要的 use case,從輸入到輸出走一遍
4. **切出模組邊界**——依據「哪些東西常常一起變」來分組
5. **定義介面與依賴方向**——盡量讓具體實作依賴抽象,不要反過來

這時候,pattern 才會自然浮現出來——不是你套上去的,是你發現「啊,這個結構剛好就是 Observer」。

### Design Pattern 的正確角色

Pattern 真正的價值有三個:

- **共通詞彙**——當你跟同事說「這裡用 Strategy」,對方馬上理解結構
- **經驗濃縮**——每個 pattern 都是前人在某類問題上踩過坑後的解法
- **事後辨識**——寫完一段程式發現「這其實就是 Factory」,可以重構得更清楚

### 常見的誤用

新手學完 Design Patterns 那本書後最容易做的事:**到處硬套 pattern**。

- 明明一個函式就解決了,硬要包成 Strategy + Factory
- 只有一個實作,卻先抽出 interface「為了未來擴展」
- 用 Singleton 包一切共享狀態,結果變成全域變數的偽裝
- 用 Observer 處理只會觸發一次的事件

這些都會讓程式變難懂、難測、難改。

### 比較實用的學習路徑

與其背 23 個 GoF pattern,不如先掌握:

**SOLID 原則**——比 pattern 更底層,幾乎所有 pattern 都是 SOLID 的某種體現。特別是 SRP(單一職責)和 DIP(依賴反轉)。

**幾個高頻 pattern**:
- Strategy——同一件事有多種做法可切換
- Factory——物件建立邏輯複雜或需要解耦
- Observer / Pub-Sub——一個變化要通知多處
- Adapter——介接不相容的介面
- Repository——資料存取抽象化
- Decorator——動態加功能而不改原本的類別

**架構層級的 pattern**(比物件層級更常用):
- MVC / MVVM / MVP——UI 與邏輯分離
- Layered Architecture——分層
- Hexagonal / Clean Architecture——把業務邏輯保護在核心
- Event-driven——非同步解耦
- CQRS——讀寫分離(只在需要時用)

### 規則

**讓 pattern 服務你的問題,不要讓你的問題去服務 pattern。**

實務上:先用最直接、最笨的方式寫出可運作的版本,讓它正確。然後問自己「這段哪裡讓我不舒服?哪裡之後改會痛?」針對那些點重構。這時候 pattern 會自己跳出來告訴你它適合在哪裡用。

## 4. 開發流程總覽

完整的開發流程有兩個層次:**單一功能的開發流程**(從想法到上線)以及**團隊協作的方法論**(Scrum、Kanban 等)。

### 單一功能的完整流程

1. **需求釐清(Requirements)** — 釐清為什麼要做、給誰用、做到哪算成功
2. **設計(Design)** — 系統設計 + 細節設計
3. **拆解與排程(Planning)** — 拆成可獨立完成、可驗證的小任務
4. **實作(Implementation)** — 小步前進、邊寫邊測
5. **測試(Testing)** — 單元、整合、E2E、效能、安全
6. **Code Review** — 不只抓 bug,還包含知識傳遞與設計把關
7. **部署(Deployment)** — CI/CD、多環境、藍綠/金絲雀策略
8. **監控與維護(Monitoring)** — 看實際運行狀況,從資料回推下一輪
9. **回顧(Retrospective)** — 每段工作後檢視,改善流程

### 團隊協作方法論

**Waterfall(瀑布式)**:需求 → 設計 → 實作 → 測試 → 部署,線性走完。適合需求極度明確、變動少的場景。發現問題時已經太晚,現在純瀑布式越來越少。

**Agile(敏捷)**:核心精神是「短週期迭代、持續調整」。不追求一次規劃到位,而是承認需求會變。底下有幾種具體做法:

- **Scrum**——固定長度的 sprint(通常 1–2 週),每個 sprint 開始時規劃、結束時 demo + retrospective。適合需求清楚但會演化的產品開發。
- **Kanban**——不分 sprint,用看板視覺化工作流程,限制同時進行中的任務數量(WIP limit)。適合任務持續流入、優先序常變的場景。
- **XP(Extreme Programming)**——強調 pair programming、TDD、持續整合、簡單設計、勇於重構。比較偏工程實踐而非流程管理,常跟 Scrum 混用。

實務上很少有團隊嚴格走某一種,通常是混合——例如「Scrum 的節奏 + Kanban 的看板 + XP 的工程實踐」。

### 核心觀念

**流程是工具,不是目的**。流程存在是為了減少溝通成本、降低風險、提高品質。如果流程反過來成為負擔(每天開會兩小時、文件比程式多),那它就壞了,該調整。

- **小團隊或個人專案**:不需要 Scrum 那一套。但仍然要有意識地走完「需求 → 設計 → 實作 → 測試 → 部署 → 回顧」這個循環,只是輕量化。
- **大團隊**:流程的價值在於讓二十個人能協調行動。Scrum/Kanban 的儀式感是必要的。

**最重要的習慣是「回顧」**——沒有回顧,流程不會進化。

## 5. 階段一:需求釐清

**負責人**:PM / BA(商業分析師)
**核心觀念**:這是整個流程的根。後續所有設計、實作、測試都以此為依據。

### 進行方式

- 與 stakeholder(出資方、業務、客服、實際使用者代表)分批訪談,而不是只跟一個人講
- 必要時做使用者研究:問卷、現場觀察、競品分析
- 草稿出來後召開**需求對齊會議**,確認大家認知一致再定稿

### 要回答的核心問題

- 這是要解決什麼問題?(不是「要做什麼功能」,是「為什麼要做」)
- 誰是使用者?他們現在怎麼解決?
- 成功的標準是什麼?怎麼知道做完了?(量化指標,不只是「好用」)
- 有什麼限制?(時間、預算、技術、法規、合規)
- 哪些是必須(must)、哪些是最好有(nice to have)?
- **這個專案不做什麼?**(明確劃出 out of scope,避免後期 scope creep)
- **有哪些假設與風險?**(假設若不成立會怎樣?)

### Review 機制

- 由 stakeholder、技術 lead、設計師一起 review
- 通過後 stakeholder 簽核,作為後續所有階段的依據
- 若後期需求變更,要回頭更新此文件並重新對齊

### 進入下一階段的標準

- Stakeholder 對「為什麼做、做什麼、做到哪」有共識
- Must / Nice to have 清楚劃分
- 成功指標可衡量

## 6. 階段二:設計

分**系統設計**與**細節設計**兩層,各自有 review 把關。

### 系統設計(System Design)

**負責人**:技術 lead / 架構師,資深工程師參與討論
**核心觀念**:探索多個方案、做 trade-off 分析,而不是直接給一個答案

#### 進行方式

- 開 1–2 次 design workshop,白板/線上協作工具一起畫
- 不是一個人關起來寫,而是團隊一起思考

#### 產出的五份文件

1. **領域模型(Domain Model)**——核心實體、它們之間的關係、業務規則
2. **模組切分(Module Decomposition)**——系統由哪些模組組成、各自職責、邊界在哪
3. **資料流(Data Flow)**——資料從哪裡來、經過哪些處理、到哪裡去
4. **介面定義(Interfaces)**——模組之間、系統與外部的介面契約(高層次)
5. **技術選型(Tech Stack)**——選了什麼技術,**重點是「為什麼選這個」與「考慮過什麼替代方案」**

#### Review 機制

- 由另一位資深工程師或架構師 review(不只是同事掛名)
- 走 **Architecture Review Meeting**,讓 reviewer 提問、挑戰假設
- Review 重點:可擴展性、可維護性、技術風險、是否過度設計、是否符合需求

#### 把關機制

- Reviewer 簽核
- 技術選型有明確理由,不是「我熟所以選這個」
- 已識別主要技術風險與緩解方案

### 細節設計(Detailed Design)

**負責人**:負責該模組的工程師
**核心觀念**:系統設計是骨架,細節設計是肉,兩者不能矛盾。發現矛盾時要回頭修系統設計。

#### 產出的四份文件

1. **資料表(Data Schema)**——表結構、欄位、型別、索引、關聯、migration 策略
2. **API 規格(API Spec)**——每個端點的路徑、方法、參數、回應、狀態碼
3. **錯誤處理(Error Handling)**——錯誤分類、錯誤碼規範、log 策略
4. **邊界條件(Edge Cases)**——空值、極大/極小值、併發、重試、超時、部分失敗

#### Review 機制

- 由另一位工程師(最好是會接觸這塊的人)review
- **跨模組介面的細設要由兩邊都 review**,避免介面雙方認知不一致

#### 把關機制

- Reviewer 簽核
- 細設與系統設計一致;若有偏離,系統設計已同步更新
- 邊界條件清單夠完整(不是只列了 happy path)

## 7. 階段三:拆解與排程

**負責人**:技術 lead + PM,實作者參與
**核心觀念**:實作者參與才會對估算負責。lead 一個人發配出來的時程沒人會珍惜。

### 進行方式

走 **planning meeting**——大家一起拆任務、估算。可用 story point、人天、或 T-shirt size(S/M/L),重點不是精準,是讓大家對「大小」有共識。

### 拆解原則

- 拆成可以獨立完成、可以驗證的小任務
- 每個任務最好 1–2 天可完成。**太大代表還沒拆夠細,通常意味著風險點沒被識別出來。**
- 每個任務有明確的「完成定義(Definition of Done)」——程式寫完不算,要包含測試、文件、review 通過
- 標出任務之間的依賴關係

### 排序原則

- 優先處理**風險高、不確定性高**的部分,而不是簡單的部分
- 把可能讓整個方案失敗的假設,放最前面驗證(spike / POC)
- 同等風險下,優先處理**能讓使用者最早拿到價值**的任務

**反模式**:不要把簡單任務放前面只為了「先看到進度」——高風險的東西拖到最後一旦炸掉,整個排程都要重排。

### 產出的文件

1. **任務拆解(Task Breakdown)**——所有任務的清單
2. **里程碑與時程(Milestones & Timeline)**——主要里程碑、預估完成日、關鍵交付物
3. **風險登記簿(Risk Register)**——已識別的風險、影響、機率、緩解策略。**這份要持續維護,不是寫完就丟。**

### 把關機制

- 所有任務都拆到 1–2 天等級
- 高風險任務排在前面
- 團隊對「做完是什麼樣子」有共識
- 主要風險已被識別並有應對方案

## 8. 階段四:實作

### 進行方式

- 採用 **feature branch + pair programming**(關鍵或高風險任務)
- **trunk-based 或 GitFlow**(視團隊規模)
- 每個任務開一條 branch,完成後走 PR 流程

### 核心實作原則

- **小步前進**:每次只改一件事,能編譯、能跑、能測。不要一次寫五百行才第一次執行
- **邊寫邊測**:不一定要嚴格 TDD,但關鍵邏輯要有測試。寫測試的過程會逼你想清楚介面
- **先求對,再求好**:第一版能動就好,確認思路正確後再重構。一開始就追求完美架構,常常是過度設計的開端
- **經常 commit**:小而頻繁的 commit,訊息寫清楚。出問題時容易回溯、容易 review
- **任何「魔法」都要註解**:為什麼這樣寫比寫了什麼更重要

### 文件與產物(每個任務)

1. **程式碼本身**——遵守事先約定的 coding style(由 linter/formatter 強制)
2. **單元測試**——覆蓋率目標
3. **任務完成筆記(Dev Notes)**——這個任務遇到什麼坑、做了什麼取捨、有哪些 follow-up
4. **更新的 API 文件 / Schema 文件**——若有改動,同步更新前面細設文件
5. **更新的 README 或模組文件**——新模組要有「這是什麼、怎麼用、怎麼跑、怎麼測」

### Review 機制

- *Code Review*:每個 PR 至少一位 reviewer(關鍵模組兩位)。Review 重點不只 bug,還有可讀性、命名、是否符合設計文件、是否有測試
- *Design 對齊檢查*:Reviewer 對照細設文件確認沒偏離。如果實作中發現設計有問題,**回頭更新設計文件**,不要讓文件與程式碼分家
- *Pair Programming*:高風險、高複雜度的任務直接兩人一起寫,省下後面 review 的來回

## 9. 階段五:測試

### 測試金字塔

| 層次 | 目標覆蓋 | 由誰寫 | 何時跑 |
|------|---------|--------|--------|
| 單元測試 | 核心邏輯 ≥ 80% | 開發者 | 每次 commit / CI |
| 整合測試 | 模組間關鍵介面 | 開發者 / QA | CI |
| 契約測試 | 服務間 API 契約 | 開發者 | CI |
| E2E 測試 | 主要 user journey | QA | 每晚 / 上線前 |
| 效能測試 | 高流量端點 | QA / SRE | 每週 / 大改前 |
| 安全測試 | OWASP Top 10 | Security | 上線前 / 季度 |
| 探索式測試 | 預期外情境 | QA | 上線前 |
| UAT | 使用者驗收 | 業務 / PM | 上線前 |

不是每個專案都需要全部都做,但要意識到哪一層你跳過了,以及它的風險。

### 環境分層

| 環境 | 用途 | 資料 | 誰用 |
|------|------|------|------|
| Local | 開發者自測 | 假資料 | Dev |
| Dev | 整合測試 | 假資料 | Dev / QA |
| Staging | 端對端 / UAT | 類 prod(去敏感化) | QA / 業務 |
| Production | 線上 | 真實 | 限唯讀 / smoke test |

### 把關機制

- 所有測試在 CI pipeline 自動跑,失敗就擋住 merge
- 上線前有一道 **QA sign-off**——QA 簽字才能進部署流程
- 重大 release 前做一次完整迴歸測試

### 缺陷分級

| 等級 | 定義 | 處理 |
|------|------|------|
| P0 | 系統不可用 / 資料損毀 / 安全漏洞 | 立即修,擋上線 |
| P1 | 主要功能無法用 / 影響大量使用者 | 上線前修 |
| P2 | 次要功能有問題 / 影響少數 | 排程修 |
| P3 | 小瑕疵 / nice-to-fix | 列入 backlog |

## 10. 階段六:Code Review 與品質把關

(這部分在後面 [Code Review 深入指南](#14-code-review-深入指南) 有完整展開。這裡先講機制。)

### 機制

- 每個 PR 必須有人 review,不允許自我 merge
- 關鍵模組(金流、權限、資料安全)要求兩位 reviewer,且其中一位是該領域的 owner
- Review 有 SLA(例如 24 小時內回覆),避免卡住開發節奏
- 定期(每月或每季)做 **architecture review**,檢查整體是否在偏離原本設計、技術債累積情況

### 產出

1. **Coding Standards 文件**——團隊共識的風格規範
2. **Review Checklist**——避免 reviewer 漏看重點
3. **技術債登記簿(Tech Debt Log)**——review 時發現但這次不修的東西,記下來,定期清理

## 11. 階段七:部署

### 環境規劃

- *Local*:開發者自己的機器
- *Dev*:整合開發環境,持續部署 main branch
- *Staging*:跟 production 設定盡量一致,QA 與 UAT(使用者驗收測試)用
- *Production*:正式環境

### 部署流程

1. PR merge 進 main → 自動觸發 CI(跑測試、build、scan)
2. 通過 → 自動部署到 Dev
3. 經 QA 驗證 → 手動或排程部署到 Staging
4. UAT 通過 + 相關 stakeholder 簽核 → 部署到 Production

### 部署策略

| 策略 | 說明 | 適合場景 |
|------|------|---------|
| **藍綠部署(Blue-Green)** | 兩套環境切換,快速回滾 | 一般情況 |
| **金絲雀發布(Canary)** | 先放給 1% / 5% / 25% / 100% 使用者 | 高風險變更 |
| **Feature Flag** | 功能上線但用開關控制,可隨時關閉 | 需要逐步開放、A/B 測試 |
| **Rolling Update** | 逐台更新 | stateless 服務 |

### 把關機制

- 部署到 Production 需要 approval(至少一位 senior 或 release manager)
- 重大變更走 **Change Advisory Board(CAB)** 或類似輕量版審核
- 每次部署後有 **smoke test** 自動跑,確認基本功能還在

### 重要文件

1. **部署手冊(Runbook)**——怎麼部署、怎麼回滾、誰有權限、緊急聯絡人
2. **Release Notes**——這次上線了什麼、修了什麼、已知問題
3. **回滾計畫(Rollback Plan)**——出問題時的具體步驟,事先寫好,不要等出事才想
4. **Infrastructure as Code**——所有環境設定用程式碼管理(Terraform、Pulumi、Ansible)

### 部署窗口

- 建議:週二~週四 10:00–16:00
- 避開週五下午、週末、節日前
- Freeze 期:雙 11、黑五、年底促銷等重大行銷活動前後

## 12. 階段八:監控與維護

**核心觀念**:上線不是結束,是另一個開始。

### 監控層次

1. **基礎設施監控**——CPU、記憶體、磁碟、網路(Prometheus、Datadog、CloudWatch)
2. **應用監控(APM)**——回應時間、錯誤率、吞吐量
3. **日誌(Logging)**——結構化日誌集中管理(ELK Stack、Loki、Splunk)
4. **追蹤(Tracing)**——分散式追蹤看請求在系統內怎麼流動
5. **業務指標**——使用者數、轉換率、功能使用率
6. **使用者體驗監控(RUM)**——真實使用者端的效能與錯誤

### 黃金訊號(Google SRE)

**最重要的 4 個指標**:

| 訊號 | 量測 |
|------|------|
| **Latency** | P50 / P99 回應時間 |
| **Traffic** | RPS(requests per second) |
| **Errors** | 5xx 比例 |
| **Saturation** | CPU / Memory / 連線池使用率 |

**重要學習**:Saturation 不只是「夠不夠」,還要看「離極限多遠」。資源類指標(連線池、thread pool、queue depth)要監控使用率,80% 預警。

### 告警(Alerting)

**告警分級**:

| 等級 | 定義 | 回應時間 | 通知方式 |
|------|------|---------|---------|
| **P0** | 全面服務中斷、資料損毀 | 立即(< 5 min) | PagerDuty 電話 + SMS |
| **P1** | 主要功能受影響 | 15 min 內 | PagerDuty 推播 + Slack |
| **P2** | 次要問題 | 工作時間內 | Slack 通知 |
| **P3** | 預警 / 趨勢警告 | 隨時 | 信箱 / dashboard |

**好告警的標準**:可行動、有意義、即時、有 context。**反模式**:Alert fatigue(假警報太多)、沒 runbook 的告警(醒了卻不知道幹嘛)、純告知性 alert(該用 dashboard 不是 alert)、對症不對因。

### SLI / SLO / SLA

- **SLI(Service Level Indicator)**——指標(例:成功請求比例)
- **SLO(Service Level Objective)**——目標(例:99.9% 可用性)
- **SLA(Service Level Agreement)**——對外承諾(通常比 SLO 寬鬆)

### Postmortem 文化

重大事故後寫 incident report,內容包含:
- 摘要、影響、時間軸
- **根本原因分析(5 Whys)**
- 哪些做得好
- 哪些可以改善
- **行動項目(必須有負責人、期限、票號)**

**核心原則:Blameless(不究責)**——對事不對人,目的是學習不是處罰。系統與流程的問題,不是個人的問題。

## 13. 階段九:回顧與迭代

**核心原則**:Action items 要有人、有期限、有追蹤——沒這三項的回顧就是聊天。

### 節奏

- **Sprint Retro**——每個 sprint 結束(1–2 週一次),團隊小範圍回顧
- **Project Retro**——專案或大功能結束,跨團隊回顧
- **Quarterly Review**——每季度高階回顧,看大方向

### Retro 流程(60 分鐘版本)

1. **暖身**(5 min)——用一個顏色形容這個 sprint、能量等級 1–10 之類的開場
2. **數據回顧**(5 min)——拿事實鋪墊,不要憑感覺
3. **收集**(15 min)——大家寫便利貼,先寫不討論。常用格式:What went well / What didn't / Questions / Ideas to try
4. **群組與排序**(10 min)——歸類後投票挑出最想討論的 3–5 個
5. **討論**(20 min)——針對票數最高的深入討論,每個 5–7 分鐘
6. **Action Items**(5 min)——**這是最重要的一段**

### Action Items 三要素

| 要素 | 內容 |
|------|------|
| **負責人** | 誰負責 |
| **期限** | 什麼時候完成 |
| **衡量成功** | 怎麼知道做到了 |

少一個都會變廢話。**每次 retro 第一件事先 review 上次 action items**。

### 主持人注意事項

**安全感**:
- Blameless,不指責個人
- 主管在場時更要注意,避免人不敢講

**引導**:
- 安靜的人主動點名問
- 講太久的人禮貌打斷
- 主持人不該主導意見,只引導

**避免**:
- 變成抱怨大會,沒有 action
- 變成只討論技術,忽略流程與協作
- Action items 太多(超過 5 個 = 一個都不會做)

### Lessons Learned 知識累積

把每次 retro / postmortem 的學習摘要進一個累積文件,新人加入時必讀。文件會走、人會換,**寫下來才會傳承**。

## 14. Code Review 深入指南

### 心態是最重要的部分

技巧再多,心態錯了就毀了。

- **Review 的是程式碼,不是人**——「這段邏輯有問題」跟「你怎麼會寫成這樣」差很多
- **假設作者已盡力**——看到奇怪的寫法,先問「為什麼這樣設計?」而不是直接斷定錯誤
- **目的是讓程式碼變好,不是證明自己懂**——Review 不是知識競賽
- **輪流當作者與 reviewer**——永遠當 reviewer 容易變得高高在上,永遠被 review 容易失去自信

### 作者要做的事(被 review 前)

很多 review 卡住的根源不是 reviewer,是作者。

**PR 要小**。500 行是上限,200 行更好。大 PR 沒人想看,看了也看不仔細。

**PR 描述要寫清楚**。至少包含:做了什麼、為什麼這樣做、怎麼測。範例格式:

```
## 為什麼(Why)
原本 X 在高並發下會超賣,因為庫存檢查與扣減不是原子操作。

## 做了什麼(What)
改用樂觀鎖,version 欄位 + 更新時帶條件。

## 怎麼測
單元測試覆蓋並發情境,壓測在 staging 跑了 1000 req/s。

## 注意
DB 需要 migration,順序:加欄位 → 部署 → 開啟新邏輯。
```

**自己先 review 一次**。開 PR 前自己用 reviewer 的視角看一遍。這 5 分鐘能省下 reviewer 半小時。

**標記重點**。如果 PR 有複雜的部分,自己留 inline comment 解釋。引導 reviewer 看你想讓他看的地方。

**主動圈人**。不要靜默地等別人發現有 PR。

### Reviewer 要做的事

**24 小時內回覆,即使是「明天看」**。卡住別人的進度比自己手上的事優先。

**先看大局,再看細節**:

1. 目的是什麼(看 PR 描述、票號、需求)
2. 整體設計(架構合理嗎?有沒有更簡單的做法?)
3. 邏輯正確性(邊界、錯誤、並發處理)
4. 可讀性(命名、結構、註解)
5. 測試(有覆蓋嗎?涵蓋邊界嗎?)
6. 細節(風格、格式、typo)

如果第 2 步就發現方向錯了,後面不用看,直接跟作者討論重做。不要從第 6 步開始挑,挑了半天最後說「啊這個設計要重來」,雙方都崩潰。

### 區分嚴重度

常見的標記方式:

| 前綴 | 意思 |
|------|------|
| `[blocking]` | 必須改才能 merge |
| `[suggestion]` | 建議改,但作者可選 |
| `[nit]` | 雞蛋裡挑骨頭,可不改 |
| `[question]` | 純粹問,沒有判斷 |
| `[praise]` | 讚賞 |

這樣作者一眼知道什麼必改、什麼可以快速回應「謝謝建議,但這次先這樣」。沒有分級時所有評論看起來都同樣嚴重,作者壓力很大。

### 用問句而不是命令句

比較:
- ❌ "改成用 map"
- ✅ "用 map 會不會更清楚?這裡 forEach 有副作用,讀起來比較容易出錯"

問句留給對方解釋的空間,可能對方有理由。命令句假設你一定對。

### 指出好的地方

Review 不只挑錯。看到漂亮的解法、周到的錯誤處理、好的命名,就說一句。`[praise]` 標記平衡了挑刺的氣氛,也讓對方知道哪些是對的方向。

### 承認不確定

看不懂的地方不要硬挑。「我對這塊不熟,這樣做的考量是什麼?」比「這寫得很怪」健康得多。

### 常見的具體技巧

**抓「驚訝」**——讀程式碼時哪裡讓你停下來想一下?那地方通常需要加註解,或重構,或拆函式。「驚訝」是 reviewer 最有價值的訊號——你是第一個讀者,你卡住了,後面所有讀者都會卡住。

**檢查不在的東西**——新手 reviewer 只看「寫了什麼」,有經驗的 reviewer 會看「沒寫什麼」:
- 錯誤情境有處理嗎?
- 邊界條件呢(空、null、極大)?
- 並發情境想過嗎?
- 測試只測了 happy path?
- 有沒有對應更新文件?
- 有沒有可能的安全問題(injection、IDOR、敏感資料 log)?

**順著資料流走**——從輸入進來,跟著資料一路走到輸出,問:每一步有沒有可能出錯?有沒有可能被惡意輸入?有沒有可能值為空?

**質疑抽象**——看到 interface 或抽象基類,問:「現在有幾個實作?」只有一個 = 過度設計,可以刪。

**質疑複雜**——看到很「聰明」的程式碼,問:「真的需要這麼複雜嗎?簡單版本有什麼問題?」聰明的代碼下一個人看不懂、改不動、debug 不出來。

**對著 diff 不夠**——重要 PR 要 checkout 下來實際跑,或至少在 IDE 裡看完整檔案,而不是只看 GitHub 上紅綠對比。

### 該抓什麼、不該抓什麼

**該抓的**(只有人能抓):
- 設計問題(架構、抽象、職責)
- 邊界與錯誤情境
- 安全問題
- 業務邏輯錯誤
- 命名是否符合領域
- 測試的完整性
- 對未來維護者是否友善

**不該抓的**(交給工具):
- 縮排、引號、空白 → formatter
- 未使用變數、命名風格 → linter
- 型別錯誤 → type checker
- 過時依賴 → dependabot

如果你還在 review 中講「這裡少一個空白」,代表工具沒配好,先把工具補上,人類去做工具做不到的事。

### 常見的壞掉模式

**Rubber stamp(橡皮圖章)**——看都不看就 LGTM。
→ 通常是 PR 太大、Reviewer 太忙、文化上 review 被當成形式。解法:設 PR 大小上限、輪換 reviewer、retro 中討論。

**Bikeshedding(雞蛋裡挑骨頭)**——只挑命名、空白、無關緊要的細節,真正的設計問題沒人看。
→ 通常是 reviewer 不熟業務、不敢碰大議題。解法:用 `[nit]` 前綴標出來,團隊一起對齊「該抓什麼」。

**Review 變辯論大會**——同一個點來回十幾條 comment。
→ 文字解不開的事直接開 15 分鐘語音/視訊。決議寫回 PR 留紀錄。

**Self-merge**——自己 approve 自己合進去。
→ 工具上禁止。除非緊急修 prod,且事後補 review。

### 業界經驗法則

- 一次 review **不超過 60 分鐘**,超過品質會掉
- 一次 review **不超過 400 行**程式碼,超過漏 bug 的機率劇增
- Review 速度 **< 500 LOC/hour**,慢慢看才看得到
- 收到 PR 到第一次回覆 **< 24 小時**

## 15. Reviewer 分配策略

### 為什麼不該固定同一個 reviewer

- **Bus factor = 1**——只有一個人懂這塊,他請假、離職、生病,專案就卡住
- **知識集中**——其他人對這塊永遠陌生,以後維護、debug、接手都得找原本那個人
- **單點偏見**——每個人都有盲點。同一個 reviewer 連續看同類程式碼,他習慣的寫法會慢慢變成「團隊唯一的寫法」
- **作者沒成長**——永遠被同一個人 review,容易養出「寫給某某看」的習慣
- **Reviewer 也累**——一個人扛全部 review 是過勞配方

### 常見的 reviewer 分配模式

**1. CODEOWNERS(GitHub / GitLab 原生功能)**

在 repo 根目錄放一個 `CODEOWNERS` 檔案:

```
/src/modules/order/        @order-team
/src/modules/order/*.sql   @order-team @dba-team
/src/modules/payment/      @payment-team @security-team
/.github/                  @platform-team
*                          @backend-leads
```

PR 改到對應路徑時,GitHub 自動 request review。Owner 通常是「團隊」而不是「個人」。

**2. 隨機 / 輪流分配**

從 reviewer pool 裡輪流派,或用 auto-assign / Slack bot 隨機指派。**好處**:強制知識輪換、避免依賴單一人。**壞處**:被分到完全不熟的領域時 review 品質有限。可搭配「至少一位熟悉者 + 一位輪換」。

**3. 雙 reviewer 策略**

關鍵 PR 要兩位 reviewer,其中一位是該領域 owner,另一位是「外部視角」:
- Owner 確保業務邏輯與架構正確
- 外部視角確保可讀性、知識傳播、抓 owner 的盲點

**4. 依 PR 類型分配**

- 安全相關 → security team 必看
- DB schema 變更 → DBA 必看
- API 變更 → API consumer team 必看
- 效能敏感 → SRE 看

**5. Pair Programming 替代 review**

對於高複雜度或高風險的任務,兩人一起寫,寫完不再正式 review。

### 怎麼選

不是非選一個,通常是組合:

| 情境 | Reviewer 策略 |
|------|--------------|
| 小團隊(3–5 人) | 全員都能 review,輪流自然發生 |
| 中型團隊(單 repo,多模組) | CODEOWNERS by 模組 + 跨模組大改要加跨團隊 reviewer |
| 大型團隊(多 repo) | CODEOWNERS by 服務 + 平台/安全/DBA 對應路徑必看 |
| 新人 onboarding | 新人 PR 由 mentor 看 + 隨機一位(讓其他人也熟悉新人) |
| 高風險 PR(安全、金流、prod 設定) | 雙 reviewer,且至少一位 senior |
| Hotfix / 緊急修 | 一位 reviewer 加速,但事後補 retrospective |

### 實務考量

- **「熟悉度」與「外部視角」要平衡**
- **Reviewer 不該都是同階級**——資淺工程師也應該 review 資深的 PR
- **避免「政治正確的批准」**——當 reviewer 是上司時,作者收到的回饋會偏軟
- **新人作為 reviewer 也要安排**——從入職第二週就開始,可以從小 PR 開始
- **Review 不該是「找誰有空」**——被分到 review 是工作的一部分,不是「順便幫忙」

### 健康團隊的指標

- 隨機抽 10 個 PR,看 reviewer 分布——是不是集中在 1–2 人?
- 每個模組是不是至少有 3 人能 review?(bus factor 的硬指標)
- 新人在加入 3 個月後,是不是已經 review 過至少 10 個 PR?
- 跨團隊 PR 的 review 來回次數,是不是比團隊內顯著多?(代表介面文件可能不夠清楚)
- Review 等待時間中位數,是不是 < 24 小時?

## 16. Reviewer 該讀什麼、讀多深

### Reviewer 需要的背景資料

從外到內,**不是每次都全讀**,依需要展開:

**第一層:PR 本身的描述**——這是最基本的。如果 PR 描述空白或只有一行,reviewer 第一個動作應該是退回去請作者補。

**第二層:對應的票號 / 需求**——PR 應該連到 issue tracker,reviewer 進去看原本要解決什麼問題、驗收標準、有沒有討論串。

**第三層:設計文件(若有)**——對於非 trivial 的改動,看對應的 API spec、Data schema、Domain model 等。重點是檢查實作有沒有偏離設計。

**第四層:相關的歷史 context**——ADR(架構決策紀錄)、過去類似 PR 的討論、相關的 incident / postmortem。

**第五層:程式碼本身的周邊**——PR 只是 diff,但被改的函式還有誰在呼叫?改的 interface 還有什麼實作?GitHub 上的 diff view 看不到這些,需要 reviewer 主動探索。

實務上時間分配大概是:**讀 PR 描述跟票號 5 分鐘,讀程式碼 20–40 分鐘**。如果讀描述跟票號就花 30 分鐘,通常代表 PR 描述太爛,該退回。

### 「讀過」的層次

不是只有「讀」跟「沒讀」兩種:

| 層次 | 做了什麼 | 適合場景 |
|------|---------|---------|
| **掃過** | 只看檔名與 diff 摘要 | 純文件、設定值小調整 |
| **讀過** | 逐行看完 diff | 大多數一般 PR |
| **理解** | 看 diff + 周邊 context | 改業務邏輯 |
| **跑過** | 本機 checkout、跑測試、實際操作 | 複雜功能、UI 改動 |
| **驗證** | 自己寫額外測試 / 嘗試破壞 | 安全相關、金流、資料一致性 |

**多數 PR 需要「理解」這層,而不是「讀過」就好**。只看 diff 經常會錯過重要的事。

### 必須完整讀的情況

- 業務邏輯改動
- 安全相關(權限、認證、加密、輸入驗證)
- 金流、資料一致性
- API 對外契約
- DB migration
- 效能敏感的熱路徑
- 基礎設施 / 部署設定

### 可以「掃過」就好的情況

- 純文件改動
- 依賴版本升級(routine)
- 大規模重構但行為不變(靠測試而不是肉眼)
- 自動產生的程式碼
- 格式化大改

### 大 PR 怎麼辦

500 行以上的 PR 在 review 上幾乎一定會出問題。處理方式:

1. **第一選擇:退回去拆分**
2. **若真的不能拆**(例如 framework 升級):
   - 分多次 review
   - 至少兩位 reviewer
   - 約一個 30 分鐘的 walkthrough,作者帶看
   - 接受「不會 100% 抓出問題」這個現實

### 不熟這塊時怎麼辦

很多人遇到不熟的程式碼就退縮。其實「不熟」反而是有價值的 review 視角:

- 老實說「我對這塊不熟,以下是我從外部視角的觀察」
- 問問題比下判斷有用
- 看可讀性——你看不懂,別人也會看不懂
- 看測試——測試清楚的話,實作通常也清楚
- 抓「驚訝」——哪裡讓你停下來想?那地方需要改進

不熟也能抓的東西:命名、函式長度、有沒有測試、錯誤處理、hardcoded secret、log 敏感資料、是否符合 coding standards。

### 推薦的 review 順序

**Step 1(2 分鐘):掃 PR 描述 + 票號**

**Step 2(3 分鐘):看 PR 結構**——改了哪些檔案?有沒有意料外的檔案被動到?PR 大小合理嗎?

**Step 3(5 分鐘):看測試**——有沒有對應的測試?涵蓋哪些情境?不熟業務時,從測試讀懂作者意圖。

**Step 4(10–30 分鐘):讀程式碼**——先看入口,順著資料流走,重點檔案展開看周邊 context。

**Step 5(視情況):本機驗證**

**Step 6:寫 comment**——區分嚴重度、用問句、指出做得好的地方。

### Reviewer 的負擔取決於作者

- PR 描述寫好 → reviewer 不用猜
- PR 拆小 → reviewer 看得仔細
- inline comment 標註複雜處 → reviewer 知道你想讓他看哪裡
- 連結相關文件 → reviewer 不用自己找
- 自己先 review 一次 → reviewer 不用幫你抓 typo

**好作者讓 reviewer 變強,壞作者讓 reviewer 變慢**。

## 17. 核心提醒

### 流程不是目的

**文件與流程是降低風險的成本,不是品質的保證**。真正讓品質高的是:清楚的思考、誠實的回顧、把人當人對待的協作文化。流程只是讓這些東西比較容易發生。

### 規模對應

- **個人 / 小專案**:不需要 Scrum、不需要 27 份文件。但仍要有意識地走完「需求 → 設計 → 實作 → 測試 → 部署 → 回顧」這個循環,只是輕量化
- **中型專案**:加上設計文件、API spec、邊界條件、風險登記簿、測試計畫、部署手冊
- **大型 / 對外產品 / 多人團隊**:全套流程

判斷標準:「不做這件事,會不會有人重複問同一個問題?或者重複踩同一個坑?」會 → 做;不會 → 跳過。

### 三類事情,維護策略不同

| 類型 | 範例 | 維護 |
|------|------|------|
| 一次性 | Requirements、Test Report、Incident Report | 寫完歸檔 |
| 活文件 | API Spec、Schema、Runbook、Risk Register | 隨程式碼一起改,過時即失效 |
| 累積文件 | Lessons Learned、ADR | 持續加,不刪除歷史 |

### 自動化 > 文件

能被工具自動檢查的,優先用工具(linter、CI、API spec 自動產生),而不是寫進文件靠人記。

### Review 不是儀式

每份文件、每個 PR 的 reviewer 都要實際讀、實際挑戰。掛名簽核比沒簽核更糟,因為它讓人誤以為被把關過了。

### Blameless 文化

Postmortem 與 Retrospective 都對事不對人。出事時人會自我保護,只有事先建立規則才能讓人敢說真話。

### 過時的文件比沒文件糟

過時的文件會誤導,定期 review。維護不動的文件就刪掉,留著只會害人。

### 持續調整

這套流程是起點,不是終點。團隊用一陣子,根據自己的痛點調整、增刪、改寫。沒有一套通用流程適合所有團隊。**最重要的習慣是「回顧」**——沒有回顧,流程不會進化。
