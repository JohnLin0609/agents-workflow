# 測試計畫(Test Plan)

> **目的**:定義「這個專案要測什麼、怎麼測、誰測、什麼狀態算測完」。
> **負責人**:QA Lead(或技術 lead 兼)
> **Review**:技術 lead、PM、開發團隊
> **時機**:在實作前或實作初期就定下,不要等開發完才想怎麼測。

---

## 1. 範圍

### 1.1 測試範圍(In Scope)
- 訂單建立、查詢、取消流程
- 金流整合(僅 happy path 與主要錯誤)
- 庫存扣減一致性
- 通知發送

### 1.2 不在範圍(Out of Scope)
- 第三方系統內部行為(信任 vendor 測試)
- 歷史資料遷移(另案處理)
- 行動版 UI(由 mobile 團隊獨立測)

## 2. 測試層次(Test Pyramid)

> 比例僅供參考,依專案調整。底層多、頂層少。

| 層次 | 目標覆蓋 | 由誰寫 | 何時跑 |
|------|---------|--------|--------|
| 單元測試 | 核心邏輯 ≥ 80% | 開發者 | 每次 commit / CI |
| 整合測試 | 模組間關鍵介面 | 開發者 / QA | CI |
| 契約測試 | 服務間 API 契約 | 開發者 | CI |
| E2E 測試 | 主要 user journey | QA | 每晚 / 上線前 |
| 效能測試 | 高流量端點 | QA / SRE | 每週 / 大改前 |
| 安全測試 | OWASP Top 10 | Security | 上線前 / 季度 |
| 探索式測試 | 預期外情境 | QA | 上線前 |
| UAT | 使用者驗收 | 業務 / PM | 上線前 |

## 3. 測試環境

| 環境 | 用途 | 資料 | 誰用 |
|------|------|------|------|
| Local | 開發者自測 | 假資料 | Dev |
| Dev | 整合測試 | 假資料 | Dev / QA |
| Staging | 端對端 / UAT | 類 prod(去敏感化) | QA / 業務 |
| Production | 線上 | 真實 | 限唯讀 / smoke test |

## 4. 工具與框架

| 用途 | 工具 |
|------|------|
| 單元測試 | Jest / Pytest / Go test |
| Mock | Sinon / unittest.mock |
| 整合 | Testcontainers |
| 契約 | Pact |
| E2E | Playwright / Cypress |
| 效能 | k6 / JMeter / Locust |
| 安全 | OWASP ZAP / Snyk |
| 覆蓋率 | Istanbul / Coverage.py |

## 5. 測試資料策略

### 5.1 來源
- 合成資料(faker 等工具)
- Prod 資料去敏感化後的子集
- 固定 fixture(關鍵情境)

### 5.2 重置策略
- 每個測試獨立、互不依賴
- 整合測試用 transaction rollback
- E2E 測試用測試專屬帳號 + 清理腳本

### 5.3 敏感資料
- **絕對不能用真實使用者資料於非 prod 環境**
- 個資、金流必須匿名化或合成

## 6. 進入標準與退出標準

### 6.1 進入測試的標準
- [ ] 程式碼通過 lint + 單元測試
- [ ] PR 已 merge 到測試 branch
- [ ] 已部署到目標環境
- [ ] 對應的 test cases 已寫好

### 6.2 上線前的退出標準
- [ ] 所有 P0/P1 test case 通過
- [ ] 已知 bug 中 P0 = 0,P1 ≤ 1(且有應變)
- [ ] 覆蓋率達成目標
- [ ] 效能達 SLO
- [ ] 安全掃描無 high/critical
- [ ] UAT 簽核

## 7. 缺陷管理

### 7.1 嚴重度分級
| 等級 | 定義 | 處理 |
|------|------|------|
| P0 | 系統不可用 / 資料損毀 / 安全漏洞 | 立即修,擋上線 |
| P1 | 主要功能無法用 / 影響大量使用者 | 上線前修 |
| P2 | 次要功能有問題 / 影響少數 | 排程修 |
| P3 | 小瑕疵 / nice-to-fix | 列入 backlog |

### 7.2 流程
- 用 issue tracker 管理
- 每個 bug 含:重現步驟、預期 vs 實際、環境、嚴重度、負責人
- 修完要有測試覆蓋,避免回歸

## 8. 自動化策略

- CI:每次 push 跑單元 + 整合
- 每晚:跑完整 E2E + 效能 baseline
- 每週:跑完整安全掃描
- 上線前:full regression

## 9. 風險與緩解

| 風險 | 緩解 |
|------|------|
| 測試環境不穩 | 加上 retry + 區分「程式錯」vs「環境錯」 |
| Flaky test | 隔離 → 修復 → 不修就刪 |
| 第三方依賴難測 | 用契約測試 + mock |
| 測試資料外洩 | 嚴格匿名化、權限控管 |
