# 技術選型(Tech Stack)

> **目的**:決定要用什麼技術,並說明「為什麼」與「考慮過的替代方案」。
> **負責人**:技術 lead / 架構師
> **Review**:資深工程師、有經驗者
> **核心原則**:每個選擇都要能回答「為什麼不是別的?」

---

## 1. 整體技術棧總覽

| 層次 | 選擇 | 主要理由 |
|------|------|---------|
| 前端 | React + TypeScript | 團隊熟悉、生態完整 |
| 行動端 | React Native | 與 Web 共用大部分邏輯 |
| 後端 | Node.js + NestJS | 與前端同語言、模組化好 |
| 資料庫 | PostgreSQL | 關聯需求多、ACID |
| 快取 | Redis | 業界標準、團隊有經驗 |
| 訊息佇列 | Kafka | 高吞吐、需要 replay |
| 雲端 | AWS | 既有合約、團隊熟悉 |
| 部署 | Kubernetes (EKS) | 規模需要 |
| CI/CD | GitHub Actions | 與 repo 整合好 |
| 監控 | Datadog | 既有合約 |

## 2. 每個選擇的詳細分析

> 每個重要決策都用以下格式記錄。可參考 ADR(Architecture Decision Record)做法。

### ADR-001:選擇 PostgreSQL 作為主資料庫

**狀態**:Accepted
**日期**:YYYY-MM-DD
**決策者**:[姓名]

**背景**
我們需要儲存訂單、使用者、庫存資料,具備強一致性、複雜查詢、交易支援。

**考慮的方案**

| 方案 | 優點 | 缺點 |
|------|------|------|
| PostgreSQL | ACID、JSON 支援、生態完整、team 熟悉 | 水平擴展較難 |
| MySQL | 廣泛使用、雲端服務多 | JSON 支援較弱 |
| MongoDB | 彈性 schema | 弱一致性、複雜 join 差 |
| DynamoDB | 雲端原生、擴展性好 | vendor lock-in、查詢受限 |

**決策**
選 PostgreSQL。理由:資料模型偏關聯、需要交易、初期規模 PostgreSQL 足夠應付,未來透過 read replica 與 sharding 可擴展。

**取捨**
- 失去:雲端原生 DB 的維運便利
- 換得:資料模型彈性與查詢能力

**重新評估時機**
- 寫入量超過 10k TPS
- 跨區複製成為瓶頸
